（Live笔记）产品汪的五种武器

2017-07-06

写在前面

​ 很早以前就开始逛知乎，但这却是我赞助的第一个知乎Live。少楠老师在Live中分享的一句让我印象比较深刻的话是“记的东西可以不多，但需要及时回顾”，“Read it later = Read it Never”。因此比较惭愧的是这场Live是我在5月4号的时候听的，却在2个月之后的现在才回顾。

​ 少楠老师分享的产品汪的五种武器分别是：记录、实现、架构、数据、整理。

灵感记录

​ 少楠老师首先提到了产品需求文档PRD（Product Requirement Document）。那么产品经理为什么要写产品需求文档，最显然的一个作用是这是一份和开发们对接需求时需要参照的一个文档。但此外产品需求文档的另外一个作用，就是记录当时思路。这一点其实我在工作中也有体会，有时候会针对产品的一个功能点和同事细究很深然后再得出结论。但如果不及时做记录，过了一段时间很容易会忘记当时的思路和结论。而将每一次讨论的结论都更新到产品需求文档里，就能起到记录的作用。

​ 灵感的来源之一是思考。每天关掉手机一段时间，给自己一个安静的环境，或许是灵感不错的来源。

​ 另外可以记录见闻，但记得需要用自己的话总结。从前看到一些好的文章我也会做一些笔记，但往往只是简单的照搬全抄，甚至是复制粘贴到Leanote，这样做其实真正能记住的东西很少。比较好的做法是看到好的观点，先简单思考过后，再用自己的话总结记录。

​ 记的东西可以不多，但需要及时回顾。

​ 可以一段时间就针对某一块知识内容做思考做探究。有感觉了做个记录就好。

​ 定期翻看和总结是关键。

快速实现

​ 这一点让我想到了在大学课堂里软件工程课上学到的敏捷开发。其核心思想就是用最快速的方法实现最基本的原型，然后再在这个基础上进行修改，不追求一开始就大而全的设计。

​ 对于设计产品也是一样，用最快的时间开发出最基本的产品让市场去检验是最好的方式，而不是在产品设计阶段就去想象这个产品最完美的形态应该是什么样。

​ 设计产品形态最好的方式就是草图，先在纸上把产品形态记录下来。

​ 速写本是个不错的工具，再配上铅笔。

​ 便利贴可以用来记录脑海中的片段，串联起来就是一个场景。

架构梳理

​ 少楠老师在这里推荐了《Web信息架构》这本书。

​ 此外，Omnigraffle是用来梳理产品信息架构的工具，可以帮助去理解整个产品的信息架构。

数据分析

​ 如果产品经理要自己做数据分析，那还是要懂一点开发的，最基本的就是SQL语句要会写。

​ 如果你要做数据分析，中位数会比平均数要更加客观。我们公司自己做的第一个产品是Minfo，是一款服务于校园的短信平台。我们会统计单个用户的平均单次通知短信发送量，但由于几个发送量较大的用户存在，这个平均数会有点被拉高。其实统计所有用户发送短信量的中位数可能会更加能反映客观事实。

​ 还有一点就是样本偏差。某些看似被反馈了很多的功能实际上可能只代表了少部分人的需求。在做Minfo的时候，我们的用户也有不少，也会陆陆续续地收到用户的反馈。我们很重视用户的意见，也会针对用户的反馈进行产品的更新。但回过头来看，其实我们当时针对用户反馈做的修改绝大部分没有带来产品体验本质上的提升，也并没有吸引来更多的用户。

思路链接

​ 每个人的潜意识都需要留出空间来建立连接。我想每个人的方式都不太一样，有些人喜欢骑自行车，骑的过程中就会不断冒出新思路。而我自己在洗热水澡的时候会觉得思路很开阔。

提问环节

  • 每做一个行业的产品都需要多去了解这个行业。了解这个行业的利益是如何流动和分配的。

    比如餐饮行业有钱的其实是连锁店，看似小而美的咖啡馆什么的并不挣钱，但是小的奶茶店却又非常暴利。理解完这些才能去设计产品。

  • 多看一些产品，多记录一些灵感

  • 少楠老师说他很喜欢贝佐斯的「十年之后什么不变」的思考方式。

    像心理学、经济行为学、统计学这些都是经历了时间考验的内容，并且是和人息息相关的，在十年之内不会有巨大的改变的。

总结

​ 第一场知乎Live带给我的收获还是很多的。虽然我在过去两个月里面并没有很刻意地去回顾这场Live中的内容。但其实过去两个月中我许多思维方式的转变都源自于这场Live。比如我现在在做笔记的时候我就会很刻意地去用自己的话总结原文的观点。也会刻意地给自己留出一定的思考的时间。我想这大概就是做分享的意义。
